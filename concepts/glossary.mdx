Short definitions for the vocabulary used across these docs. Each links to the page where the concept is developed in full.

## Identity & keys

- **`did:key`** ‚Äî A [W3C DID](https://www.w3.org/TR/did-core/) that encodes an Ed25519 public key directly in the identifier (multicodec `0xed01` + the 32-byte key), so the identifier *is* the key and signatures verify offline ‚Äî no registry, no DNS. The only agent-account method in v0.1. ‚Üí [Identity and keys](/concepts/identity-and-keys)
- **`agent_did`** ‚Äî An agent's `did:key`, used as the signing `keyid` on every request and the `sub` of its attestations. It changes when the key rotates. ‚Üí [Identity and keys](/concepts/identity-and-keys)
- **`account_id`** ‚Äî A service-local, opaque account identifier, distinct from any key and **stable across key rotation**, so one account can hold several agent credentials. ‚Üí [How it works](/how-it-works)
- **`service_did`** ‚Äî The service's own DID, declared in its discovery document (typically `did:web`, since services have stable hosts) and used as the `aud` an attestation is bound to. ‚Üí [Identity and keys](/concepts/identity-and-keys)
- **`did:web`** ‚Äî A DNS-anchored DID method. **Not** an agent identity in AFAuth (agents have no stable web origin); used only for a service's `service_did` and for `did`-type owner recipients. ‚Üí [Identity and keys](/concepts/identity-and-keys)

## Signing & verification

- **RFC 9421** ‚Äî HTTP Message Signatures, the standard AFAuth signs requests with, so existing tooling interoperates. ‚Üí [Signing requests](/concepts/signing)
- **`content-digest`** ‚Äî A covered component carrying the SHA-256 of the request body (RFC 9530); included whenever there's a body, so tampering breaks verification. ‚Üí [Signing requests](/concepts/signing)
- **`nonce`** ‚Äî A unique per-request signature parameter; the verifier keeps a seen-set of `(keyid, nonce)` tuples and rejects reuse with `replayed_nonce`. ‚Üí [Signing requests](/concepts/signing)
- **`created` / `expires`** ‚Äî Signature timestamps; `expires ‚àí created` is capped at 300 seconds, verified with ¬±60 s of skew tolerance. ‚Üí [Signing requests](/concepts/signing)
- **Freshness window** ‚Äî The interval a signature is valid and its nonce must be remembered (`expires ‚àí created + skew`). ‚Üí [Signing requests](/concepts/signing)

## Attestation & trust

- **Attestation** ‚Äî A JWT, presented in the `AFAuth-Attestation` header, in which an attestor vouches for the agent's runtime ‚Äî by default, that a verified human stands behind it. ‚Üí [Attestation](/concepts/attestation)
- **Attestor** ‚Äî The party that signs attestations. AFAuth recognizes four classes: trust, platform, commerce, and service-operator HMAC. ‚Üí [Attestation](/concepts/attestation)
- **Trust attestor** ‚Äî The canonical `afauth-trust` issuer at [trust.afauth.org](https://trust.afauth.org), which mints short-lived, audience-bound JWTs that an agent is bound to a human ‚Äî verifiable offline against its JWKS. ‚Üí [The trust attestor](/concepts/trust-attestor)
- **`sub_h`** ‚Äî A per-service, pairwise human pseudonym (HMAC-SHA256, ‚â• 128 bits, opaque, no PII), stable per `(human, service)` and unlinkable across services ‚Äî the handle services bucket per-human quotas and bans on. It is also the OIDC `sub` when the human signs in, so attestation and [Sign in with AFAuth](/concepts/human-oidc-signin) converge on one account. ‚Üí [The trust attestor](/concepts/trust-attestor)
- **Binding (trust link)** ‚Äî The link between an agent DID and a human account at the trust attestor (`afauth trust link`). Transfers no ownership, creates no account, lasts ~90 days, and lets the agent mint attestations. ‚Üí [The trust attestor](/concepts/trust-attestor)
- **`unclaimed_mode`** ‚Äî The discovery field declaring how a service treats pre-claim signups: `attested_only`, `free`, or `denied`. ‚Üí [Attestation](/concepts/attestation)
- **`attested_only`** ‚Äî The `defineService` default: implicit signup is refused unless a valid attestation is present (`401 attestation_required`, no account created). ‚Üí [Attestation](/concepts/attestation)
- **`free` (mode)** ‚Äî A service that doesn't require attestation; a bare signed request suffices. ‚Üí [Attestation](/concepts/attestation)
- **`verification` (claim)** ‚Äî The trust-attestation claim naming the strongest method the human has on file: `email`, `oauth`, or `payment`. Services rank it by their own policy; unknown values MUST be ignored, not rejected. ‚Üí [The trust attestor](/concepts/trust-attestor)
- **Attested session** ‚Äî Keeping a fresh attestation on file and re-challenging when it lapses (¬ß10.7), so an attestor-side revoke reaches signature-gated services within the window. ‚Üí [Keep attested access live](/guides/keep-attested-access-live)
- **JWKS / JWT** ‚Äî The JSON Web Key Set a service verifies attestation JWTs against (the trust attestor publishes its keys at `trust.afauth.org/.well-known/jwks.json`), and the JSON Web Token an attestation takes. ‚Üí [Attestation](/concepts/attestation)
- **Sign in with AFAuth** ‚Äî Human social-login: a person signs in via the trust attestor (an OpenID Provider) and lands in the `(iss, sub_h)` account their agent already created. Authentication, not ownership. ‚Üí [Sign in with AFAuth](/concepts/human-oidc-signin)
- **`id_token`** ‚Äî The OIDC token the trust attestor mints at sign-in: `sub` is the same pairwise `sub_h` as the agent's attestation, `iss` is the URL `https://trust.afauth.org`, `aud` is the service's `service_did`; it carries no PII. ‚Üí [Sign in with AFAuth](/concepts/human-oidc-signin)

## Ownership & lifecycle

- **Ceremony** ‚Äî The owner invitation + claim flow that hands an account from agent to human, gated by the invariant that the agent's signature alone can't bind ownership. The security boundary of AFAuth. ‚Üí [The ceremony](/concepts/ceremony)
- **Claim** ‚Äî A human authenticating *as* the invited recipient at the service's claim page, which (after the match relation) moves the account to `CLAIMED`. ‚Üí [The ceremony](/concepts/ceremony)
- **Two-step verify** ‚Äî The ceremony's two independent authentications ‚Äî the agent's signature and the human's authentication ‚Äî that together enforce the invariant. ‚Üí [The ceremony](/concepts/ceremony)
- **Match relation** ‚Äî The rule a service applies at claim time to confirm the human's authenticated identity equals the staged recipient before transitioning to `CLAIMED`. ‚Üí [The ceremony](/concepts/ceremony)
- **Recipient** ‚Äî The identity an agent stages in an invitation. The spec defines `email`, `phone`, `oidc`, and `did`; the v0.1 reference SDK ships only the `email` handler. ‚Üí [Invite and claim](/guides/invite-and-claim)
- **Account states** ‚Äî `UNCLAIMED` (created, no owner) ‚Üí `INVITED` (invitation pending) ‚Üí `CLAIMED` (owned); plus `EXPIRED` (only on services that opt into an `unclaimed_ttl_seconds`; absent by default) and `ARCHIVED` (owner-deleted). ‚Üí [The ceremony](/concepts/ceremony)
- **Owner-binding operation** ‚Äî A change to which credentials can authenticate as the owner; requires a *freshly* authenticated owner session (60‚Äì300 s), never the agent key alone. ‚Üí [Invite and claim](/guides/invite-and-claim)

## Operations

- **Revocation** ‚Äî Cutting an agent off via two independent levers: **global** (revoke the binding at the attestor ‚Äî stops new attestations everywhere) and **local** (revoke or re-key at one service). ‚Üí [Revocation](/concepts/revocation)
- **Error envelope** ‚Äî The single shape every conformant error takes: `{ error: { code, message, details? } }`, where `code` is a stable identifier clients program against. ‚Üí [Error envelope](/concepts/error-envelope)
- **Service directory** ‚Äî The opt-in, non-normative registry of AFAuth services at [registry.afauth.org](https://registry.afauth.org); a listing proves control of the discovery host, not endorsement. ‚Üí [Service directory](/concepts/service-directory)
