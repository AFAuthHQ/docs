An AFAuth account is named by a **Decentralized Identifier** ([W3C DID](https://www.w3.org/TR/did-core/)). Two methods are defined for v0.1: `did:key` (default) and `did:web` (recommended for long-lived accounts).

<Note>
The v0.1 **reference** SDK and Worker resolve `did:key` only — in-process, no
network. `did:web` is defined by the spec and recognised in the SDK's types,
but resolution is an opt-in hook the reference Worker does not ship ([ADR-0003](https://github.com/AFAuthHQ/spec/blob/main/implementation/adr/0003-did-key-only-resolver.md)).
The `did:web` guidance below describes the protocol-level design.
</Note>

## `did:key`

A `did:key` identifier encodes the public key directly into the identifier:

```
did:key:z6MkiYbwC5honA2sxE7XLAyJMDFibLvVg8FgodBX4A4CaUgr
```

The multibase string after `did:key:` is the multicodec prefix for Ed25519 (`0xed01`) followed by the 32-byte raw public key, base58btc-encoded. A service verifying a `did:key` signature decodes the string locally and uses the recovered public key directly — **no registry lookup, no DNS**. The identifier *is* the key.

Tradeoffs:

- **Verifiable offline.** Anyone can verify a `did:key` signature with just the DID string and the signature itself.
- **Cheap to generate.** `Agent.generate()` in the TypeScript SDK takes microseconds.
- **Tightly coupled to the key.** If the agent rotates its key, the account identifier changes — external references to the old DID stop resolving. This makes `did:key` a poor fit for accounts that need to persist across rotation.

## `did:web`

A `did:web` identifier looks like a hostname:

```
did:web:agent.example.com
```

Resolution maps the DID to an HTTPS URL: `did:web:agent.example.com` → `https://agent.example.com/.well-known/did.json`. The DID document at that URL lists the agent's verification keys.

Tradeoffs:

- **Stable across rotation.** The identifier doesn't encode any specific key, so rotating publishes a new key in the DID document without changing the DID.
- **Anchored in DNS + TLS.** Trust is rooted in the same machinery as the rest of the web — for services, this is usually preferred over self-issued public keys with no authority anchor.
- **Requires a host.** You need to be able to serve `https://<host>/.well-known/did.json`, which means `did:web` is appropriate for agents that have a stable web presence.

The protocol recommends `did:web` for long-lived agents. See [§3.1.2](https://github.com/AFAuthHQ/spec/blob/main/spec/core.md#312-didweb) for the full normative resolution rules.

## Key handling

The agent's private key is the **sole credential** for pre-claim operations. There's no recovery path other than re-creating the account.

Storage recommendations, in increasing order of robustness:

| Storage | When it's enough | When it isn't |
|---|---|---|
| File on disk (mode `0600`) | Local development, scripts | Anywhere an attacker might read the FS |
| OS keystore (Keychain, Credential Manager, Secret Service) | Single-user agents | Multi-user or server deployments |
| TPM / Secure Enclave / HSM | Production agents on managed hosts | — |
| Cloud KMS with signing-only access | Server-side agents at scale | — |

The reference CLI stores the active key at `~/.afauth/key.json` with mode `0600`. The TypeScript SDK lets you supply your own seed via `Agent.fromPrivateKey(seed)` so you can plug into any of the above.

## Rotation

Pre-claim rotation is a one-line operation:

```typescript
const newAgent = await rotateAgentKey(oldAgent);
// See examples/recipes/rotate.ts
```

The agent signs the rotation request with the **old** key. The service revokes the old DID and accepts the new one going forward. Post-claim rotation requires owner approval — see [§8](https://github.com/AFAuthHQ/spec/blob/main/spec/core.md#8-key-management).

For the runnable example: [`examples/recipes/rotate.ts`](https://github.com/AFAuthHQ/typescript-sdk/blob/main/examples/recipes/rotate.ts).

## Per-service keys

By default, an agent that signs up to multiple services with the same `did:key` is correlatable across those services. Agents that require unlinkability MUST derive per-service keys — the spec ([§3.3](https://github.com/AFAuthHQ/spec/blob/main/spec/core.md#3-identity)) suggests HKDF-based derivation from a master seed. The v0.1 SDK does not ship a per-service derivation helper; bring your own or generate fresh agents per service.
